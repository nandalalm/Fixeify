FIXEIFY DTO MAPPING REFACTOR PLAN
=================================

Goal
----
Create one consistent DTO mapping architecture across the backend:

Controller -> Service -> Mapper -> Response DTO
                      |
                      -> Repository -> MongoDB

Repositories must handle persistence and return typed documents/query records.
Services must handle business logic and use mappers to produce response DTOs.
Controllers must handle HTTP concerns only.
DTO files must define API request/response shapes and must not depend on Mongoose models.

MAPPING RULES
-------------
[x] A mapper accepts a strongly typed model, lean record, populated record, or
    aggregation result and returns a response DTO.
[x] A mapper is a pure function with no database access or side effects.
[x] Repository methods return:
    - Mongoose documents for writes and simple entity reads.
    - Typed lean/populated records for read views.
    - Typed aggregate records for aggregation queries.
    - Primitive totals/statistics where appropriate.
[x] Services call repositories, apply business logic, and invoke mappers.
[x] Controllers send service results without remapping domain entities.
[x] Request DTO-to-persistence conversion belongs in a named input mapper or
    service command builder, never in the response mapper.
[x] DTO constructors, if retained, accept explicit typed properties only.
[x] DTO constructors must not inspect `unknown`, Mongoose population state, or
    raw database documents.
